## PHASE 2 — CTO AUDIT

### Architecture & Module Boundaries
- **Status**: Good. Unidirectional data flow is respected (UI -> Service -> DB).
- **Critique**: Sales and Purchasing act as orchestrators triggering Inventory and Finance. This is acceptable, but risks circular dependencies if not strictly monitored.

### Database & Transaction Integrity
- **Status**: At Risk.
- **Critique**: The transaction boundary (where `session.commit()` and `session.rollback()` occur) is currently managed inside the UI layer (e.g., `sales_widget.py`). This dangerously skirts *Law 8: UI Never Owns Business Logic*. If a UI developer forgets to wrap a service call in a `try/except` with a rollback, the database session could become tainted.

### Audit Coverage
- **Status**: Functional, but fragile.
- **Critique**: Audit logs are generated by manually placing `AuditService.record_event()` calls inside Service methods. This violates *Law 4 (Audit Everything)* by introducing the human element: a developer *will* eventually forget to add it to a new function.

### Permission Coverage
- **Status**: Hardcoded.
- **Critique**: `PermissionManager` works via `RequestContext`, but roles are hardcoded upon login. To scale, this needs to be a database-driven `RolePermission` mapping.

### Testing Coverage
- **Status**: Poor (15%).
- **Critique**: Only `InventoryService` has a proper `pytest` suite enforcing CRUD, Authorization, Validation, and Rollbacks. Other modules rely entirely on the fact that their architecture "matches" Inventory.

### UI Consistency & Performance
- **Status**: Clean, but unscalable.
- **Critique**: Using `QTableWidget` is excellent for rapid delivery but will consume massive CPU/Memory allocations for companies with >10,000 invoices. *Law 20* demands long-term scalability.

## PHASE 5 — IMPROVEMENT PROPOSALS

Before implementing new features, I propose the following architectural pivots:

1. **Introduce a Unit of Work (UoW) Context Manager**:
   Move `session.commit()` and `session.rollback()` out of the PySide6 widgets and into the Service layer or a dedicated `@transactional` decorator. This permanently eliminates the risk of UI-induced database locks.
2. **Implement SQLAlchemy Event Listeners for Audit**:
   Remove `AuditService.record_event` from the business logic. Instead, bind to SQLAlchemy's `after_insert`, `after_update`, and `after_delete` events to automatically serialize `before_values` and `after_values` to the `AuditEvent` table. This guarantees 100% audit coverage automatically.
3. **Introduce a Base UI Widget**:
   `CustomerWidget` and `SupplierWidget` share 95% of their layout and logic. Abstracting a `BaseDataWidget` will reduce UI code duplication and make future upgrades (like moving to `QTableView`) much easier.

## PHASE 6 — SELF CRITIQUE

As the architect, I must identify my own shortcuts taken for the sake of delivery speed:

- **Mistake**: Allowing the UI widgets to dictate when a database transaction commits. This is a severe violation of Enterprise patterns.
- **Assumption**: Assuming developers will always remember to write `AuditService.record_event()` when adding new endpoints. This is technically naive.
- **Shortcut**: `AuthenticationService.login` hardcodes permissions into a Python `set`. A real company needs to configure custom roles dynamically in the database.
- **Enterprise Weakness**: The complete lack of automated testing for `SalesService` and `PurchasingService`, despite them orchestrating the most critical atomic operations in the company (Money & Stock).

## PHASE 7 — EXECUTION PLAN

The following roadmap replaces immediate feature development with enterprise hardening tasks to prepare for v1.0.

### T-0019: Transaction Boundary Hardening
- **Objective**: Move `commit()` and `rollback()` out of UI widgets into a Service-level Unit of Work / Decorator.
- **DoD**: UI widgets no longer reference `session.commit()`.

### T-0020: Automated ORM Audit Trailing
- **Objective**: Refactor Audit generation to use SQLAlchemy event listeners.
- **DoD**: Manual `record_event` calls removed; tests prove updates trigger audit records automatically.

### T-0021: Core Service Test Suite Completion
- **Objective**: Expand the `InventoryService` `pytest` standard to `SalesService`, `PurchasingService`, and `FinanceService`.
- **DoD**: 100% line coverage for critical path atomic transactions.

### T-0022: Dynamic Role-Based Access Control (RBAC)
- **Objective**: Create `Role` and `Permission` database tables to replace the hardcoded Python sets in `AuthenticationService`.
- **DoD**: UI allows Administrators to dynamically toggle granular permissions for Employee profiles.

### T-0023: UI Scalability Pivot (QTableView)
- **Objective**: Abstract a `BaseDataModel` (`QAbstractTableModel`) and refactor all widgets to use `QTableView` with lazy-loading chunking to support 100,000+ rows.
- **DoD**: Memory consumption remains stable when navigating between views with heavy data.
