# Database Migration Policy

**Version:** 1.0
**Status:** IMMUTABLE

This document governs the permanent rules for database evolution in TAJ FROID ERP starting from Release Candidate 1 (RC1).

## 1. Single Source of Truth
Alembic is the **ONLY** authority allowed to modify the database schema.
- `BaseModel.metadata.create_all()` is **FORBIDDEN** in the main application.
- `BaseModel.metadata.drop_all()` is **FORBIDDEN** in the main application.
- Manual SQL schema modifications are **FORBIDDEN**.
- Runtime table creations are **FORBIDDEN**.

## 2. Immutability of Migrations
Existing migration scripts in `migrations/versions/` must **NEVER** be edited.
If a mistake is discovered, do not edit history. Create a new migration to resolve it.

## 3. Human Verification Policy
Autogenerated migrations (`alembic revision --autogenerate`) are NOT trusted by default.
Every migration must be manually reviewed before commit to verify:
- No unexpected `drop_table` or `drop_column`.
- Indexes are preserved.
- Constraints are correct.
- Operations are compatible with SQLite.

## 4. Failure Abort Protocol
If an Alembic migration fails during application startup:
1. The application logs the error explicitly.
2. The user is presented with a safe failure dialog.
3. The application aborts startup via `sys.exit(1)`.
**The application must NEVER continue with an unknown schema state.**

## 5. Pre-Migration Safety Protocol
Before any migration is executed, the application will create a timestamped backup of the `.db` file in the `backups/` directory.
If the backup copy operation fails, the migration must abort to prevent data loss.
